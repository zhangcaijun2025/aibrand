## 二、关键讨论与决策

### 决策 1：五层架构定版
- **讨论**：组件职责混乱，谁做中枢？
- **你的方案**：OpenClaw L5 通信网关 → Dify L4 AI应用 → LangChain L3 编排 → n8n L2 自动化 → Claude L1 研发
- **DeepSeek 建议**：n8n 应该做中枢调度，不应放在底层。异步回调优先，统一消息 Schema。
- **融合结论**：n8n 升级为 L2 中枢调度层 — 唯一有权编排跨组件任务。外部只触达 Dify(人) 或 n8n(系统)。LangChain→Dify 工具调用属豁免范围。

### 决策 2：刚性红线
- 每层有明确的"必做"和"禁止"清单
- LangChain Agent 禁止互相调用（防递归爆炸）
- 组件间禁止共享数据库直连、禁止本地文件硬耦合
- 统一 REST API + X-Request-ID 全链路追踪

### 决策 3：异步优先
- >30s 任务必须走 webhook 回调，避免 HTTP 阻塞
- n8n 是唯一回调接收者
- 统一消息 Schema：{task_id, intent, payload, context, callback_webhook}

### 决策 4：模型分层计费
- Dify 内容生成 → Haiku (30% 预算)
- LangChain 推理 → Sonnet (40%)
- Claude 代码审计 → Opus (20%)
- OpenClaw 路由 → Haiku (10%)

### 决策 5：Docker 数据卷
- **问题**：Windows bind mount 导致 SQLITE_READONLY
- **解决**：n8n 改用 Docker 命名卷
- **规则**：所有需要写入的容器优先使用命名卷

### 决策 6：自进化系统
- **你的愿景**：系统每天自动进化、改进、成长，同步甚至超越时代前沿
- **我的设计**：L1自愈→L2自优化→L3自进化→L4自创造 四级模型
- **你的补充**：用户自适应层 — 越用越懂用户，自动生成 Skill 和工作流
- **结论**：今天建成 L1，明天推进 L2

## 五、关键经验与改进

### 做得好的
- 多智能体并行执行（4 Agent 同时跑），效率提升 3-4x
- 三方审阅融合（你 + DeepSeek + 我），产出远超单方
- 快速试错不僵持（n8n SQLite 问题→转向命名卷）
- 全链路代码落地，每个组件到可运行状态

### 需要改进的
- n8n webhook CLI 激活浪费 40min → 规则：3次失败即切 Web UI
- LangChain 版本未提前锁定 → 固定 requirements.txt
- 统一 Schema 后置导致返工 → Phase 0 就应定义
- Docker 网络超时无预案 → 准备离线镜像包

### 沟通模式
- 你提供架构方向 + 业务需求 → 我负责技术实现 + 工程落地
- DeepSeek 提供独立审阅 → 取长补短融合
- OpenClaw 负责通知 + 归档 + IMA 知识沉淀
- 飞书/企微双通道保持信息同步
