## 1. 需求分析

### 1.1 目标用户画像

| 用户类型 | 核心需求 | 预算敏感度 | 技术要求 |
|----------|---------|:----------:|:--------:|
| **个人开发者/极客** | 学习、实验、个人助理 | 高（<$5000） | 中 |
| **小型团队（3-10人）** | 内部知识库、自动化流程、客服 | 中（<$20000） | 中 |
| **中小企业（10-50人）** | 业务系统集成、客户管理、数据分析 | 较低（<$50000） | 高 |

### 1.2 核心痛点与解决方案

| 痛点 | 方案 |
|------|------|
| 数据隐私担忧，不敢用公有云AI | ✅ 全栈私有化部署，数据不出内网 |
| API调用成本高，长期使用贵 | ✅ 本地大模型推理，一次投入长期使用 |
| 现有系统与AI割裂 | ✅ N8N + Dify 打通300+应用集成 |
| 非技术人员不会用 | ✅ 低代码/零代码平台，拖拽式搭建 |
| 多模型切换麻烦 | ✅ Dify 统一模型网关，一键切换 |
| 看不见AI产出价值 | ✅ AiToEarn 赋能商业化变现 |

### 架构设计原则

| 原则 | 说明 |
|------|------|
| **模块化** | 每个组件独立容器化，可单独升级/替换 |
| **松耦合** | 模块间通过 API/Webhook 通信，不相互依赖 |
| **统一网络** | 所有容器接入 `ai-platform` 网络，容器名即地址 |
| **安全隔离** | Nginx 反向代理统一对外暴露，内部服务不直接暴露 |
| **可观测** | 健康监控中心 + 统一日志，7×24 状态可见 |
| **渐进式** | 从核心模块开始，按需扩展，避免"大而全"陷阱 |

### 9.3 Dify Agent vs PraisonAI Agent 分工

| 维度 | Dify Agent | PraisonAI Agent |
|------|-----------|----------------|
| **定位** | 轻量单Agent / 低代码 | 多Agent协作 / 复杂编排 |

| **开发门槛** | 零代码/低代码 | 需编写Agent配置文件 |
| **集成方式** | 内置插件 | 可被 Dify 工具节点调用 |

> **建议分工**：Dify 处理 C 端对话场景，PraisonAI 处理 B 端复杂分析任务
> **集成方式**：Dify 通过 HTTP 节点调用 PraisonAI API
